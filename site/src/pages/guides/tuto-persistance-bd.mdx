C'est la persistance **durable** par excellence : <Highlight>les données survivent aux redémarrages</Highlight>, se partagent entre tous les serveurs, et surtout se laissent **interroger**. Le prix à payer : il faut **modéliser**, accepter la **latence** d'un accès réseau, et écrire du code **asynchrone** (`await` partout).

## MongoDB — base documentaire

MongoDB range des **documents** (BSON, proche du JSON) souples, qu'on imbrique ou qu'on référence. Avec Mongoose, le CRUD est concis :

<Consideration tech="mongodb" />

```js
const emprunt = await Emprunt.create({ emprunteur, livreId, date: new Date() });
const histo   = await Emprunt.find({ emprunteur }).sort({ date: -1 });
```

<Highlight couleur="ambre">Indexez les champs que vous filtrez</Highlight> (`emprunteur`), sans quoi chaque lecture parcourt toute la collection. La décision *imbriquer vs référencer* détermine le coût des requêtes.

<MarginNote type="lien" titre="Modéliser un document">
La page [Prise en main de MongoDB](/notes/mongodb-prise-en-main) et son explorateur de modèle montrent l'effet des choix d'imbrication sur le coût des endpoints.
</MarginNote>

## PostgreSQL — base relationnelle

PostgreSQL range des **lignes** dans des tables au schéma fixe, avec types et contraintes. Il ajoute ce que le documentaire ne donne pas nativement : **transactions ACID** et **jointures**.

<Consideration tech="postgresql" />

```sql
INSERT INTO emprunts (emprunteur, livre_id) VALUES ($1, $2) RETURNING *;
SELECT * FROM emprunts WHERE emprunteur = $1 ORDER BY date DESC;
```

Le schéma plus rigide se paie en **migrations** quand le modèle évolue, mais il garantit des **invariants** forts (clés étrangères, contraintes) — précieux dès que les données sont très liées.

## Lequel choisir

Documentaire ou relationnel n'est pas une question de mode :

- **MongoDB** brille pour des données semi-structurées, évolutives, lues surtout **par agrégats** (on récupère un document avec ce qu'il contient).
- **PostgreSQL** brille quand les données sont très **relationnelles**, qu'on a besoin de **transactions multi-tables** et d'invariants forts.

<Checkpoint question="Un système de réservation doit garantir qu'on ne réserve jamais deux fois la même place, même sous forte concurrence. Quelle famille de base privilégier, et grâce à quoi ?">
Une base **relationnelle** (PostgreSQL) : ses **transactions ACID** et ses **contraintes** (unicité, clés étrangères) garantissent qu'une place ne peut pas être attribuée deux fois, même quand plusieurs requêtes arrivent en même temps. On peut faire de la cohérence forte ailleurs, mais le relationnel l'offre de série, ce qui réduit le risque d'erreur.
</Checkpoint>
