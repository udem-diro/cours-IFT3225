## Le pipeline, en une image

`find` filtre et renvoie des documents **tels quels**. Dès qu'on veut **calculer** sur un ensemble — compter, regrouper, faire des moyennes, joindre une autre collection — on passe au **pipeline d'agrégation** : une suite d'**étapes** où chaque étape reçoit un flux de documents, le transforme, et passe le résultat à la suivante, comme un tapis roulant.

```js
db.evenements.aggregate([ etape1, etape2, etape3 ]);
```

Chaque étape est un objet `{ $operateur: … }`. L'ordre compte : ce qui sort d'une étape est l'entrée de la suivante.

<MarginNote type="astuce" titre="Filtrer tôt">
Placez `$match` (et `$sort` sur un champ indexé) **le plus tôt possible** : moins de documents circulent dans les étapes suivantes, et MongoDB peut s'appuyer sur un index. Un `$match` en fin de pipeline travaille sur tout l'ensemble — gaspillage.
</MarginNote>

## Les étapes, une à une

On travaille sur trois collections du réseau Lino : `evenements` (`{ boite_id, livre_id, type, date }`), `boites` (`{ nom, quartier }`) et `livres` (`{ titre, categorie, annee }`).

### `$match` — filtrer

Le même document de requête que `find`, avec les mêmes opérateurs :

```js
{ $match: { type: "emprunt", date: { $gte: ISODate("2026-05-01") } } }
```

### `$group` — regrouper et calculer

`$group` produit **un document par valeur distincte** de la clé `_id`. Les autres champs sont des **accumulateurs**, qui agrègent les documents de chaque groupe :

```js
{
  $group: {
    _id: "$boite_id",            // une ligne par boîte distincte
    emprunts: { $sum: 1 },       // compter les documents du groupe
    dernier:  { $max: "$date" }, // la date la plus récente du groupe
  },
}
```

Le `$` devant `boite_id` signifie *la valeur du champ*. `_id: null` regroupe **tout** en une seule ligne (un total global).

| Accumulateur | Calcule |
|---|---|
| `$sum` | une somme (`$sum: 1` compte) |
| `$avg` | une moyenne |
| `$min` / `$max` | le plus petit / grand |
| `$push` | la liste des valeurs du groupe |
| `$addToSet` | la liste **sans doublons** |
| `$first` / `$last` | la première / dernière valeur (selon le tri en amont) |

### `$sort`, `$skip`, `$limit` — ordonner et paginer

```js
{ $sort: { emprunts: -1 } },   // -1 décroissant, 1 croissant
{ $skip: 0 },                  // sauter N résultats
{ $limit: 5 },                 // n'en garder que 5
```

### `$project` et `$addFields` — choisir et remodeler

`$project` décide des champs **sortants** (`1` garder, `0` exclure) et peut en **calculer** de nouveaux ; `$addFields` ajoute sans rien retirer :

```js
{ $project: { _id: 0, boite: "$_id", emprunts: 1 } }  // renommer _id en "boite"
```

### `$unwind` — déplier un tableau

`$unwind` transforme **chaque élément** d'un champ-tableau en un document à part. Indispensable après un `$lookup` (qui renvoie un tableau), ou pour traiter un tableau imbriqué :

```js
{ $unwind: "$livre" }
```

### `$count` — compter le flux

```js
{ $count: "total" }   // remplace le flux par { total: <nombre> }
```

<Checkpoint question="Pourquoi placer $match avant $group plutôt qu'après ?">
`$match` en tête réduit le nombre de documents **avant** le regroupement : `$group` ne brasse que les documents pertinents, et `$match` peut exploiter un index. Placé après, il faudrait d'abord regrouper *tout* l'ensemble, puis jeter une partie du résultat — le même résultat, mais bien plus de travail.
</Checkpoint>

## Joindre des collections liées : `$lookup`

Quand une donnée est **référencée** — un `ObjectId` qui pointe vers une autre collection — plutôt qu'imbriquée, il faut aller la chercher. `$lookup` est l'équivalent du `JOIN` relationnel : pour chaque document, il rapatrie les documents correspondants d'une autre collection.

```js
// Pour chaque événement, attacher le livre référencé par livre_id
db.evenements.aggregate([
  {
    $lookup: {
      from: "livres",          // collection à joindre
      localField: "livre_id",  // le champ local (la référence)
      foreignField: "_id",     // le champ correspondant dans "livres"
      as: "livre",             // résultat déposé ici (toujours un tableau)
    },
  },
  { $unwind: "$livre" },                          // une seule correspondance : on aplatit
  { $project: { _id: 0, date: 1, type: 1, titre: "$livre.titre" } },
]);
```

`$lookup` dépose **toujours un tableau** dans `livre`, car une jointure peut en théorie ramener plusieurs documents. Comme la référence pointe ici vers **un** livre, `$unwind` réduit ce tableau d'un élément à un simple objet, qu'on lit ensuite avec `$livre.titre`.

<Highlight couleur="ambre">`$lookup` est exactement le coût d'avoir *référencé* plutôt qu'imbriqué.</Highlight> Si chaque événement portait déjà son livre (imbriqué), le titre serait là sans jointure — au prix de la duplication. Le modèle décide donc si lire des données liées coûte un `$lookup`… ou rien.

<MarginNote type="attention" titre="`from` est un nom de collection">
Le `from` de `$lookup` désigne la **collection** telle qu'elle existe dans la base (`"livres"`, `"boites"`), pas le nom du modèle Mongoose (`Livre`, `Boite`). Mongoose nomme par défaut la collection en minuscules au pluriel du modèle : `Boite` → `boites`. Se tromper de nom donne un tableau vide, sans erreur.
</MarginNote>

## Composer une requête complexe, pas à pas

Objectif : **les 5 boîtes les plus actives depuis le 1er mai 2026, avec leur nom et leur quartier**. On empile les étapes une à une.

D'abord, ne garder que les emprunts de la période :

```js
{ $match: { type: "emprunt", date: { $gte: ISODate("2026-05-01") } } }
```

Compter par boîte, puis classer et tronquer :

```js
{ $group: { _id: "$boite_id", emprunts: { $sum: 1 } } },
{ $sort:  { emprunts: -1 } },
{ $limit: 5 }
```

À ce stade on a `{ _id: <boite_id>, emprunts: N }`. Il manque le nom de la boîte : on le joint, on aplatit, on remodèle. Le pipeline complet :

```js
db.evenements.aggregate([
  { $match:  { type: "emprunt", date: { $gte: ISODate("2026-05-01") } } },
  { $group:  { _id: "$boite_id", emprunts: { $sum: 1 } } },
  { $sort:   { emprunts: -1 } },
  { $limit:  5 },
  { $lookup: { from: "boites", localField: "_id", foreignField: "_id", as: "boite" } },
  { $unwind: "$boite" },
  { $project: { _id: 0, boite: "$boite.nom", quartier: "$boite.quartier", emprunts: 1 } },
]);
```

Deux variantes sur le même modèle :

```js
// Les 5 livres les plus empruntés, avec leur titre
db.evenements.aggregate([
  { $match:  { type: "emprunt" } },
  { $group:  { _id: "$livre_id", emprunts: { $sum: 1 } } },
  { $sort:   { emprunts: -1 } },
  { $limit:  5 },
  { $lookup: { from: "livres", localField: "_id", foreignField: "_id", as: "livre" } },
  { $unwind: "$livre" },
  { $project: { _id: 0, titre: "$livre.titre", emprunts: 1 } },
]);

// La répartition des emprunts par quartier : joindre AVANT de regrouper
db.evenements.aggregate([
  { $match:  { type: "emprunt" } },
  { $lookup: { from: "boites", localField: "boite_id", foreignField: "_id", as: "boite" } },
  { $unwind: "$boite" },
  { $group:  { _id: "$boite.quartier", emprunts: { $sum: 1 } } },
  { $sort:   { emprunts: -1 } },
]);
```

<MarginNote type="rappel" titre="L'ordre raconte l'intention">
Compter *puis* joindre (1er exemple) : on agrège par boîte, puis on décore le résultat. Joindre *puis* compter (3e exemple) : on a besoin du quartier — qui vit sur `boites` — *avant* de pouvoir regrouper dessus. La question « de quoi ai-je besoin pour grouper ? » fixe la place du `$lookup`.
</MarginNote>

<Checkpoint question="Pour regrouper les emprunts par quartier, faut-il le $lookup avant ou après le $group ?">
**Avant.** Le quartier n'est pas sur l'événement, il est sur la boîte. Il faut donc d'abord joindre `boites` (et `$unwind`) pour disposer du champ `boite.quartier`, *ensuite* `$group` peut l'utiliser comme clé `_id`. Joindre après le regroupement serait trop tard : la clé n'existerait pas encore.
</Checkpoint>

## Réalisation avec Express et Mongoose

Le même pipeline tourne dans une application via Mongoose : `Model.aggregate(pipeline)` renvoie une **promesse** de résultats. On part du modèle `Evenement`, qui référence les deux autres :

```js
import mongoose from "mongoose";

const evenementSchema = new mongoose.Schema({
  boite_id: { type: mongoose.Schema.Types.ObjectId, ref: "Boite" },
  livre_id: { type: mongoose.Schema.Types.ObjectId, ref: "Livre" },
  type: { type: String, enum: ["emprunt", "depot"] },
  date: { type: Date, default: Date.now },
});

export const Evenement = mongoose.model("Evenement", evenementSchema); // collection "evenements"
```

Une route de statistiques bâtit son pipeline **à partir des paramètres de requête**, l'exécute, et renvoie du JSON :

```js
import express from "express";
import { Evenement } from "../models/evenement.js";

const router = express.Router();

// GET /api/stats/boites-actives?depuis=2026-05-01&limite=5
router.get("/stats/boites-actives", async (req, res) => {
  try {
    const depuis = new Date(req.query.depuis ?? "2026-01-01");
    const limite = Math.min(Number(req.query.limite ?? 5), 50); // borner l'entrée

    const resultats = await Evenement.aggregate([
      { $match:  { type: "emprunt", date: { $gte: depuis } } },
      { $group:  { _id: "$boite_id", emprunts: { $sum: 1 } } },
      { $sort:   { emprunts: -1 } },
      { $limit:  limite },
      { $lookup: { from: "boites", localField: "_id", foreignField: "_id", as: "boite" } },
      { $unwind: "$boite" },
      { $project: { _id: 0, boite: "$boite.nom", quartier: "$boite.quartier", emprunts: 1 } },
    ]);

    res.json(resultats);
  } catch (err) {
    res.status(500).json({ erreur: "Agrégation impossible" });
  }
});

export default router;
```

Ce qui se joue dans cette route :

- **Les paramètres pilotent le pipeline.** `depuis` et `limite` viennent de `req.query` et sont injectés dans `$match` et `$limit`. La pagination s'ajoute de même avec `$skip: (page - 1) * limite`.
- <Highlight couleur="ambre">On **caste et borne** toujours les entrées avant de les insérer dans le pipeline.</Highlight> `new Date(...)`, `Number(...)`, un plafond sur `limite` : ne jamais glisser un objet brut venu du client directement dans une étape.
- **`aggregate` renvoie des objets bruts**, pas des documents Mongoose : ni getters, ni virtuels, ni méthodes d'instance. C'est voulu — on veut une forme prête à sérialiser, façonnée par `$project`.

<MarginNote type="astuce" titre="`populate` ou `$lookup` ?">
Pour résoudre une simple référence sur un résultat de `find`, Mongoose offre `Boite.find().populate("...")` — plus court. Mais `populate` agit *après coup*, document par document : il ne peut pas regrouper ni calculer. Dès qu'on **agrège** (compter, moyenner, grouper sur un champ joint), c'est `$lookup`, dans le pipeline, qu'il faut.
</MarginNote>

<MarginNote type="attention" titre="Indexer ce qu'on filtre et joint">
Une agrégation lente vient presque toujours d'un `$match` ou d'un `$lookup` sans index. Indexez `date` et `type` (le `$match`), et le `foreignField` du `$lookup` — ici `_id` l'est déjà. Sur de gros volumes, c'est la différence entre quelques millisecondes et un balayage complet.
</MarginNote>

<Checkpoint question="Pourquoi ne pas passer req.query.limite directement à $limit ?">
`req.query.limite` est une **chaîne** non vérifiée. `$limit` attend un nombre ; et une valeur démesurée (ou un objet injecté) ferait travailler la base inutilement, voire ouvrirait une faille. On la convertit (`Number`) et on la borne (`Math.min(..., 50)`) avant de l'insérer. Règle générale : toute entrée client est castée et validée avant d'entrer dans un pipeline.
</Checkpoint>

## Ce qu'on retient

- Le **pipeline** est une suite d'étapes chaînées ; on **filtre tôt** (`$match`, `$sort` indexé) pour limiter le flux.
- `$group` + **accumulateurs** (`$sum`, `$avg`, `$push`…) calcule sur des ensembles ; `$project`/`$addFields` remodèlent la sortie.
- `$lookup` **joint** une collection référencée (le `JOIN`) et renvoie un tableau qu'on aplatit avec `$unwind` — c'est le coût d'avoir référencé. Joindre **avant** de grouper quand la clé de groupe vit dans la collection jointe.
- Avec Mongoose, `Model.aggregate(pipeline)` renvoie des **objets bruts** ; le `from` du `$lookup` est un **nom de collection**, pas de modèle.
- Dans une route, on **bâtit le pipeline à partir de `req.query`** (casté, validé, borné), on pagine avec `$skip`/`$limit`, et on **indexe** les champs filtrés et joints.

<MarginNote type="plusloin" titre="Aller plus loin">
Le pipeline compte bien d'autres étapes (`$facet`, `$bucket`, `$dateToString` pour grouper par jour, `$out`…). Le [manuel d'agrégation MongoDB](https://www.mongodb.com/docs/manual/aggregation/) en donne le catalogue complet.
</MarginNote>
