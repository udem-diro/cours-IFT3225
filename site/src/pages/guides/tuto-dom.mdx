Ce guide suppose une familiarité avec HTML (balises, attributs) et les bases de JavaScript (variables, fonctions, objets). On y fait le pont entre les deux.
</MarginNote>

## Qu'est-ce que le DOM ?

Quand un navigateur charge une page HTML, il ne garde pas le texte HTML tel quel. Il le lit et en construit une **représentation en mémoire** : une structure d'objets que JavaScript peut lire et modifier. Cette structure, c'est le **DOM**.

<MarginNote type="definition" titre="DOM">
*Document Object Model* — une représentation en mémoire de la page sous forme d'arbre d'objets. Chaque balise HTML devient un **nœud** (*node*) que JavaScript peut inspecter, modifier, créer ou supprimer.
</MarginNote>

Une fois la page chargée, on ne manipule plus le HTML — on manipule le DOM. C'est cette distinction qui permet à une page de changer sans être rechargée : modifier le DOM met à jour ce que l'utilisateur voit, instantanément.

### Le pont entre deux mondes

On a vu JavaScript comme langage (variables, fonctions, objets) et HTML comme structure de page. Le DOM est <Highlight couleur="ambre">l'interface qui relie les deux</Highlight> : il expose chaque élément HTML comme un objet JavaScript, avec des propriétés (son texte, ses attributs, ses enfants) et des méthodes (le modifier, lui ajouter un écouteur d'événement).

## L'arbre du document

Le mot « arbre » n'est pas une métaphore vague — c'est une structure précise. Chaque élément a **un parent** et peut avoir **des enfants**. Le tout descend d'une racine unique (`<html>`), exactement comme une arborescence de fichiers descend d'un dossier racine.

Prenons une petite carte affichant une boîte à livres. À gauche, ce que l'utilisateur voit. À droite, l'arbre DOM que le navigateur a construit. **Survolez un élément d'un côté — son jumeau s'illumine de l'autre.** Cliquez sur un nœud pour voir ses détails.

<DomInspector client:visible />

Ce que cet inspecteur montre, c'est exactement ce que fait l'onglet **Elements** (ou **Inspecteur**) des outils de développement de votre navigateur. <Highlight>Chaque chose visible à l'écran correspond à un nœud dans l'arbre</Highlight> — et inversement. Apprendre à naviguer entre les deux vues est la compétence de base du développement web.

<MarginNote type="astuce" titre="Ouvrez vos DevTools">
Dans n'importe quel navigateur : clic droit sur un élément → « Inspecter » (ou `F12`). Vous verrez le vrai arbre DOM de la page, exactement comme dans l'inspecteur ci-dessus, mais pour la page entière.
</MarginNote>

### Vocabulaire de l'arbre

- **Nœud (node)** — tout élément de l'arbre. Une balise, mais aussi le texte à l'intérieur, ou un commentaire.
- **Élément (element)** — un nœud qui est une balise HTML (`<p>`, `<div>`, `<a>`).
- **Parent** — le nœud qui contient directement un autre nœud.
- **Enfant (child)** — un nœud contenu directement dans un autre.
- **Frères (siblings)** — des nœuds qui partagent le même parent.
- **Racine (root)** — le sommet de l'arbre, l'élément `<html>`.

<Checkpoint question="Dans la carte ci-dessus, quel est le parent des trois <li> ? Et leurs frères ?">
Le parent des trois `<li>` est le `<ul>` (la liste des tags). Les trois `<li>` sont *frères* entre eux — ils partagent le même parent. Le `<ul>`, lui, est l'enfant de `<article>` et le frère de `<h2>`, `<p>` et `<a>`.
</Checkpoint>

## Sélectionner des éléments

Pour modifier un élément, il faut d'abord l'**atteindre** depuis JavaScript. Le DOM offre des méthodes de sélection. Les deux modernes, qu'on utilise presque toujours, prennent un **sélecteur CSS** :

```js
// Le premier élément qui correspond au sélecteur CSS
const titre = document.querySelector(".carte__titre");

// Tous les éléments qui correspondent (une NodeList)
const tags = document.querySelectorAll(".carte__tags li");
```

<MarginNote type="astuce" titre="Le sélecteur CSS, partout">
Si vous savez écrire un sélecteur CSS (`.classe`, `#id`, `article > p`, `a[href]`), vous savez sélectionner dans le DOM. `querySelector` accepte exactement la même syntaxe que vos feuilles de style.
</MarginNote>

Il existe des méthodes plus anciennes, encore très répandues dans le code existant :

```js
document.getElementById("menu");           // par id (sans le #)
document.getElementsByClassName("carte");  // par classe (collection live)
document.getElementsByTagName("li");       // par balise
```

<Highlight couleur="ambre">Dans du code nouveau, préférez `querySelector` et `querySelectorAll`</Highlight> — une seule syntaxe à retenir, et la même que CSS.

### `querySelector` vs `querySelectorAll`

| | Retourne | Si rien ne correspond |
|---|---|---|
| `querySelector(sel)` | le **premier** élément (ou `null`) | `null` |
| `querySelectorAll(sel)` | **tous** les éléments (une `NodeList`) | `NodeList` vide |

Une `NodeList` ressemble à un tableau et s'itère avec `for...of` ou `forEach`, mais ce n'en est pas tout à fait un. Pour utiliser `map` ou `filter`, on la convertit d'abord : `Array.from(tags)` ou `[...tags]`.

<Checkpoint question="Pourquoi const x = document.querySelector('.inexistant') ne plante-t-il pas, mais x.textContent plante-t-il ?">
`querySelector` retourne `null` quand aucun élément ne correspond — ça ne lève pas d'erreur. Mais `null.textContent` plante, car `null` n'a pas de propriété `textContent`. C'est l'erreur `Cannot read properties of null` la plus fréquente en DOM. Toujours vérifier que la sélection a réussi avant d'utiliser le résultat, surtout si l'élément pourrait ne pas exister.
</Checkpoint>

## Lire et modifier un élément

Une fois un élément sélectionné, on accède à ses propriétés comme à n'importe quel objet JavaScript.

### Le contenu

```js
const titre = document.querySelector(".carte__titre");

titre.textContent = "Nouvelle boîte";     // remplace le texte
titre.textContent;                         // lit le texte actuel
```

<MarginNote type="erreur" titre="textContent vs innerHTML">
`textContent` traite tout comme du texte brut — sûr. `innerHTML` interprète le contenu comme du HTML — pratique, mais **dangereux** avec des données venant de l'utilisateur : c'est la porte d'entrée des attaques XSS (injection de scripts). Par défaut, utilisez `textContent`.
</MarginNote>

### Les attributs

```js
const lien = document.querySelector(".carte__lien");

lien.getAttribute("href");              // lire un attribut
lien.setAttribute("href", "/boites/7"); // modifier un attribut
lien.href = "/boites/7";                // raccourci pour les attributs courants
```

### Les classes CSS

La propriété `classList` est la façon idiomatique de manipuler les classes — donc l'apparence :

```js
const carte = document.querySelector(".carte");

carte.classList.add("active");        // ajouter une classe
carte.classList.remove("active");     // retirer
carte.classList.toggle("active");     // basculer (ajoute si absente, retire si présente)
carte.classList.contains("active");   // tester (true / false)
```

<Highlight>Modifier les classes plutôt que les styles directement</Highlight> garde la logique d'apparence dans le CSS, où elle appartient. JavaScript décide *quand* une classe s'applique ; CSS décide *à quoi elle ressemble*.

## Créer et insérer des éléments

On peut fabriquer de nouveaux nœuds et les greffer dans l'arbre :

```js
// 1. Créer l'élément
const nouveauTag = document.createElement("li");
nouveauTag.textContent = "poésie";

// 2. L'insérer dans le DOM
const liste = document.querySelector(".carte__tags");
liste.appendChild(nouveauTag);        // ajoute à la fin des enfants
```

Tant qu'un élément créé n'est pas inséré dans l'arbre, il existe en mémoire mais <Highlight couleur="violet">n'apparaît nulle part à l'écran</Highlight> — il flotte, détaché. C'est l'insertion (`appendChild`, `prepend`, `before`, `after`) qui le rend visible.

Pour retirer un élément :

```js
nouveauTag.remove();                  // se retire lui-même de l'arbre
```

<Checkpoint question="On a fait document.createElement('p') et défini son textContent, mais rien n'apparaît. Pourquoi ?">
Parce qu'on a créé l'élément en mémoire mais qu'on ne l'a jamais *inséré* dans l'arbre du document. Un élément créé est détaché : il faut l'attacher à un parent déjà présent dans le DOM avec `appendChild`, `prepend`, `before` ou `after` pour qu'il devienne visible. Créer ≠ insérer.
</Checkpoint>

## Réagir aux événements

Le DOM ne sert pas qu'à modifier la page — il permet de **réagir** à ce que fait l'utilisateur : clics, frappe au clavier, survol, soumission de formulaire. On attache un *écouteur d'événement* (*event listener*) à un élément.

```js
const bouton = document.querySelector(".carte__lien");

bouton.addEventListener("click", (evenement) => {
  evenement.preventDefault();          // empêche le comportement par défaut
  console.log("La boîte a été cliquée");
});
```

`addEventListener` prend deux arguments : le **type** d'événement (`"click"`, `"input"`, `"submit"`, `"mouseover"`...) et une **fonction** à exécuter quand il survient. Cette fonction reçoit un objet `evenement` décrivant ce qui s'est passé.

<MarginNote type="definition" titre="Événement">
Un signal émis par le navigateur quand quelque chose se produit : un clic, une touche pressée, le chargement de la page. Le code qui « écoute » cet événement (le *callback*) s'exécute en réaction.
</MarginNote>

C'est ici que tout se rejoint : <Highlight>une page interactive est une boucle — l'utilisateur agit, un événement se déclenche, JavaScript modifie le DOM, l'affichage change.</Highlight>

```js
// Exemple complet : un compteur de clics
const bouton = document.querySelector("#compteur");
let clics = 0;

bouton.addEventListener("click", () => {
  clics++;
  bouton.textContent = `Cliqué ${clics} fois`;
});
```

<Checkpoint question="À quoi sert evenement.preventDefault() dans le clic sur un lien <a> ?">
Un lien `<a href="...">` a un comportement *par défaut* : naviguer vers l'URL. `preventDefault()` annule ce comportement, ce qui permet d'exécuter du JavaScript à la place (par exemple afficher du contenu sans changer de page). Sans lui, le navigateur suivrait le lien et quitterait la page avant que le reste du code ne s'exécute.
</Checkpoint>

## Le DOM dans ce cours

On reverra le DOM sous plusieurs angles au fil de la session. Côté client, c'est la cible de toute interactivité. Mais une idée clé du cours est que <Highlight couleur="ambre">le serveur peut aussi produire du HTML prêt à insérer dans le DOM</Highlight> — c'est l'approche qu'on explorera avec HTMX, où le serveur renvoie des fragments de DOM plutôt que du JSON.

Comprendre que la page *est* un arbre manipulable — et non un texte figé — est le socle de tout ce qui suit.
