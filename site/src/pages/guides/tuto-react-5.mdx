## Les enjeux des applications complexes

| Enjeu | Réponse |
|---|---|
| Plusieurs écrans, une navigation | Décomposer en **vues (pages)** et router avec **React Router** |
| Un état partagé entre des composants éloignés | Le fournir par **React Context**, sans le faire descendre en props partout |
| Une structure qui tient à mesure que le code grandit | Découper par responsabilité : vues, composants, contexte, services |
| Un public de plusieurs langues | Externaliser les textes et basculer la langue avec **react-i18next** |

Voici l'application Bus en vue, notre fil rouge, enrichie au fil de la page : ses vues sont routées, leur état est partagé par un contexte, et ses libellés se traduisent. N
aviguez entre les vues, marquez un arrêt en favori, votez sur un signalement ; l'engrenage en haut à droite ouvre les réglages (langue, thème, taille du texte).

## Décomposer en vues (pages) et React Router

Une application complexe se pense d'abord en **vues** : des écrans distincts, chacun responsable d'une tâche. Dans Bus en vue, les anciens onglets deviennent des vues routées : **Signaler**, **Rechercher**, et **Mes arrêts** (les arrêts mis en favori). Une vue de **détail d'un arrêt** s'y ajoute, atteinte par une URL paramétrée.

**React Router** associe une **URL** à une vue. On déclare les routes une fois, et la navigation se fait par des liens, sans rechargement de page.

```jsx
import { BrowserRouter, Routes, Route, Link, useParams } from "react-router-dom";

function Application() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Signaler</Link>
        <Link to="/rechercher">Rechercher</Link>
        <Link to="/mes-arrets">Mes arrêts</Link>
      </nav>
      <Routes>
        <Route path="/" element={<VueSignaler />} />
        <Route path="/rechercher" element={<VueRechercher />} />
        <Route path="/mes-arrets" element={<VueMesArrets />} />
        <Route path="/arret/:arretId" element={<VueArret />} />
      </Routes>
    </BrowserRouter>
  );
}
```

Le segment `:arretId` est un **paramètre d'URL**. La vue le lit avec `useParams` :

```jsx
function VueArret() {
  const { arretId } = useParams();
  // ... afficher les signalements de cet arrêt
}
```

Pour changer de vue par programme (après l'envoi d'un signalement, par exemple), on utilise `useNavigate` :

```jsx
import { useNavigate } from "react-router-dom";

const navigate = useNavigate();
navigate("/"); // retour à la liste
```

<MarginNote type="attention" titre="MemoryRouter dans la page">
L'application ci-dessus est **embarquée** dans cette note. Si elle utilisait `BrowserRouter`, sa navigation modifierait l'URL de la page entière. Elle utilise donc `MemoryRouter`, qui garde l'historique **en mémoire**, à l'intérieur de l'île. Dans une vraie application autonome, vous utilisez `BrowserRouter`.
</MarginNote>

## Un état partagé avec React Context

Dans Bus en vue, une même donnée sert **plusieurs vues** : la liste des signalements s'affiche dans Signaler et Rechercher, se filtre dans le détail d'un arrêt ; les favoris pilotent « Mes arrêts » ; les votes s'appliquent partout. Avec le routage, ces vues sont rendues par `<Route element={...} />`, et leur faire descendre l'état en props devient particulièrement pénible : on ne contrôle pas les niveaux intermédiaires. C'est le *prop drilling*.

**React Context** fournit une valeur à tout un sous-arbre, que n'importe quel descendant lit directement. On crée le contexte, un **fournisseur** qui détient l'état et les actions, et un **hook** d'accès.

```jsx
import { createContext, useContext, useState } from "react";

const AppContext = createContext(null);

function FournisseurApp({ children }) {
  const [signalements, setSignalements] = useState(/* ... */);
  const [favoris, setFavoris] = useState([]);

  const signaler = (s) => setSignalements((prev) => [s, ...prev]);
  const basculerFavori = (id) =>
    setFavoris((prev) => (prev.includes(id) ? prev.filter((x) => x !== id) : [...prev, id]));
  const voter = (id, dir) => { /* met à jour up / down */ };

  const valeur = { signalements, favoris, signaler, basculerFavori, voter };
  return <AppContext.Provider value={valeur}>{children}</AppContext.Provider>;
}
```

On enveloppe un **hook personnalisé** autour du contexte, avec une erreur claire si on oublie le fournisseur :

```jsx
function useApp() {
  const ctx = useContext(AppContext);
  if (!ctx) throw new Error("useApp doit être utilisé dans <FournisseurApp>.");
  return ctx;
}
```

Le fournisseur enveloppe l'application, au-dessus du routeur :

```jsx
export default function BusSignalApp() {
  return (
    <FournisseurApp>
      <MemoryRouter initialEntries={["/"]}>
        <Coquille />
      </MemoryRouter>
    </FournisseurApp>
  );
}
```

Chaque vue lit alors l'état directement, sans recevoir une seule prop :

```jsx
function VueMesArrets() {
  const { favoris, signalements } = useApp();
  // ... afficher les arrêts favoris et leurs signalements
}
```

<Highlight couleur="ambre">Le contexte sert à un état **largement partagé** (les signalements, les favoris, les votes). L'état **local** d'une vue, lui, reste en `useState` : dans la vue Signaler, l'arrêt, la ligne et le délai en cours de sélection n'intéressent personne d'autre, ils n'ont aucune raison de monter dans le contexte.</Highlight>

## Multilingue avec react-i18next

Bus en vue sert son interface en français et en anglais. Le principe : on ne met pas les textes en dur dans les composants, on les **externalise** dans un dictionnaire par langue et chaque composant demande la traduction d'une **clé** ; un sélecteur bascule la langue et toutes les vues se retraduisent. L'instance i18next est propre à l'île (`i18next.createInstance()`), fournie par un `<I18nextProvider>`.

<MarginNote type="plusloin">
L'internationalisation est traitée en détail dans [S'adapter à l'utilisateur](react-personnalisation), avec react-i18next, l'interpolation, le formatage culturel (`Intl`) et les autres préférences de l'utilisateur (thème, taille du texte, animations).
</MarginNote>

## Structurer le projet pour la maintenance et l'évolution

Les trois piliers suggèrent un découpage où **chaque responsabilité vit à un seul endroit**. L'application embarquée tient en un fichier pour être insérée dans cette page, mais un vrai projet l'éclate ainsi :

```text
src/
├── App.jsx              // fournisseurs (i18n, contexte) au-dessus du routeur
├── routes.jsx           // table des routes (chemin vers vue)
├── pages/               // une vue par écran routé
│   ├── VueSignaler.jsx
│   ├── VueRechercher.jsx
│   ├── VueMesArrets.jsx
│   └── VueArret.jsx
├── components/          // briques d'affichage réutilisables, sans logique métier
│   ├── Coquille.jsx
│   ├── Signalement.jsx
│   ├── Vote.jsx
│   └── EtoileFavori.jsx
├── context/             // état partagé
│   ├── AppContext.js    // createContext + hook useApp
│   └── FournisseurApp.jsx
├── hooks/               // hooks personnalisés (useLocalStorage, useClickOutside)
├── services/            // accès aux données
│   └── api.js           // fetch des signalements, isole le réseau
├── i18n/                // internationalisation
│   ├── index.js         // instance i18next
│   └── locales/
│       ├── fr.json
│       └── en.json
└── data/                // constantes et données de démonstration
```

Chaque dossier porte une responsabilité claire :

| Responsabilité | Où elle vit |
|---|---|
| Orchestrer un écran | une vue dans `pages/` |
| Afficher une brique réutilisable | un composant dans `components/` |
| Détenir l'état partagé | le contexte dans `context/` |
| Parler au réseau | un service dans `services/` |
| Réutiliser une logique | un hook dans `hooks/` |
| Gérer les textes et les langues | `i18n/` et `locales/` |

<MarginNote type="rappel" titre="Vue contre composant">
Une **vue** (page) orchestre un écran : elle lit l'état et compose des briques. Un **composant** est une brique d'affichage réutilisable, sans connaissance du métier (`Signalement`, `Vote`). Cette séparation garde les vues lisibles et les composants réutilisables.
</MarginNote>

Ce découpage paie surtout au moment de faire évoluer l'application :

- **Ajouter un écran** : une vue dans `pages/`, une route dans la table des routes, un lien dans la navigation.
- **Ajouter une langue** : un fichier `locales/xx.json`, déclaré dans `i18n/`. Aucun composant ne change.
- **Brancher le vrai serveur** : remplacer `services/api.js`. Les vues lisent l'état par `useApp`, pas le réseau ; elles ne changent pas.
- **Mémoriser une préférence** (la langue, les favoris) : un hook `useLocalStorage`, sans toucher au reste.

<Highlight couleur="ambre">La règle qui tient tout : une responsabilité à un seul endroit. Un libellé se change dans une locale, une route dans la table des routes, la source de données dans un service. Aucune modification ne se disperse dans le code.</Highlight>

<Checkpoint question="Vous devez brancher l'application sur un vrai serveur de signalements, au lieu des données de démonstration. Quels fichiers changez-vous, et lesquels restent intacts ?">
On change le **service** (`services/api.js`) pour qu'il interroge le serveur, et le **fournisseur** (`FournisseurApp`) pour qu'il appelle ce service plutôt que les données de démonstration. Tout le reste reste intact : les vues et les composants lisent l'état par `useApp` sans savoir d'où il vient, et les traductions ne dépendent pas de la source des données. C'est précisément ce que permet la séparation des responsabilités.
</Checkpoint>

## Récapitulatif

| Notion | Rôle | Outils clés |
|---|---|---|
| Vues et navigation | Découper l'application en écrans liés à des URL | `BrowserRouter`, `Routes`, `Route`, `Link`, `useParams`, `useNavigate` |
| État partagé | Fournir une donnée centrale sans prop drilling | `createContext`, `Provider`, `useContext`, hook `useSignalements` |
| Multilingue | Externaliser les textes et basculer la langue | `i18next`, `useTranslation`, `t`, `changeLanguage` |
| Structure | Une responsabilité par dossier, pour la maintenance et l'évolution | `pages/`, `components/`, `context/`, `services/`, `i18n/` |

<Checkpoint question="Pourquoi l'application de cette page utilise-t-elle MemoryRouter plutôt que BrowserRouter ?">
Parce qu'elle est embarquée dans la page. `BrowserRouter` lit et modifie l'URL du navigateur ; la navigation de l'île changerait donc l'adresse de la page entière et entrerait en conflit avec le site. `MemoryRouter` conserve l'historique de navigation en mémoire, à l'intérieur de l'île, sans toucher à l'URL. Dans une application autonome, on utilise `BrowserRouter`.
</Checkpoint>
