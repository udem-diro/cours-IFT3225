## Lino — un exemple de bout en bout

Lino est un réseau de boîtes à livres en libre-service disséminées dans une ville. Des citoyens y déposent et prennent des livres librement. L'application aide à localiser les boîtes, suivre les livres, et mettre en relation les lecteurs.

On va construire l'API de Lino en suivant la démarche suivante : partir des questions que les utilisateurs posent, en déduire les endpoints, puis concevoir le modèle de données qui les supporte.

---

### Les gens d'abord

Avant de penser aux endpoints, il faut comprendre ce que les gens font *sans* l'application.

Un citoyen marche dans son quartier. Il tombe sur une boîte à livres. Il feuillette, prend un roman, laisse un essai. Il ne connaît pas le propriétaire de la boîte. Il n'a pas de compte. Il ne signe rien. La confiance est implicite — c'est un acte communautaire.

Un autre citoyen cherche un livre précis. Il fait le tour des boîtes de son quartier à pied. Parfois il trouve, souvent il ne trouve pas. Il n'a aucun moyen de savoir si le livre est quelque part dans une boîte à 3 rues de chez lui.

L'application ne doit pas *remplacer* ces pratiques — elle doit les *faciliter* sans les alourdir.

## Modèle de données

Les endpoints et les flux déterminent le modèle — pas l'inverse. La question est : quelles lectures sont les plus fréquentes?

### Collections MongoDB

```javascript
  ]

  statut: "accepté",
  rendez_vous: "Café Olimpico, 16h",
  date: ISODate("2026-05-13T10:00:00Z")
}
```

### Pourquoi des événements séparés?

Les livres dans une boîte pourraient être stockés *dans* le document de la boîte (un tableau `livres`). Mais chaque livre a un cycle de vie indépendant — il traverse plusieurs boîtes. Si les livres sont imbriqués, le parcours d'un livre (`GET /livres/QR-7890`) demande de chercher dans *tous* les documents boîtes.

En séparant les événements, le parcours d'un livre est une simple requête :

```javascript
db.evenements.find({ code_qr: "QR-7890" }).sort({ date: 1 })
```

Les livres *actuellement* dans une boîte se déduisent des événements : les dépôts sans retrait correspondant. C'est un calcul, pas un stockage — et c'est un choix de conception (stocker l'état courant vs le recalculer à partir de l'historique).

### Index

```javascript
db.boites.createIndex({ position: "2dsphere" });        // recherche géospatiale
db.evenements.createIndex({ code_qr: 1, date: 1 });     // parcours d'un livre
db.evenements.createIndex({ boite_id: 1, type: 1 });     // livres dans une boîte
db.recherches.createIndex({ titre: 1, statut: 1 });      // correspondances lors d'un dépôt
```

## TODO pour la suite

Plusieurs décisions restent ouvertes :

- **Authentification** — qui peut déposer, signaler, proposer un échange? Lino repose sur la confiance communautaire, mais les échanges impliquent un contact entre personnes.
- **Pagination** — `GET /boites/:id/livres` retourne tout? Les 20 plus récents?
- **Notifications push** — Sophie doit faire un `GET /recherches/rech-5` pour vérifier. Pour une vraie notification instantanée, il faudrait du push (SSE, WebSocket).
- **Modération** — que faire des faux signalements? Seuil automatique vs modération manuelle?
- **Vie privée des échanges** — Nadia et Léa échangent un lieu de rendez-vous via l'API. Ces données doivent être protégées.



## Ce qu'on n'a pas résolu
 
Concevoir, c'est aussi savoir ce qu'on laisse de côté — et le dire.
 
La **notification de Sophie** se fait par polling : elle doit revenir demander. Un vrai système la *préviendrait*. Mais le push (Server-Sent Events, WebSocket) est un autre modèle de communication, qui sort du cadre requête-réponse. C'est une limite assumée, pas un oubli.
 
La **modération** repose sur la confiance communautaire. Que faire si quelqu'un signale faussement qu'une boîte a disparu? On pourrait exiger un seuil de signalements concordants avant d'agir — mais sur-sécuriser un système fondé sur la confiance, c'est en trahir l'esprit. C'est une décision de conception, pas un bug à corriger.
 
L'**authentification** est minimale. Lino marche parce que les gens se font confiance. Ajouter trop de contrôle découragerait la contribution. Le bon niveau de friction n'est pas zéro, mais il n'est pas maximal non plus.
 
Aucune de ces questions n'a de réponse universelle. C'est précisément pour ça qu'elles méritent d'être documentées plutôt que tranchées en silence dans le code.

## Pour aller plus loin
 
Quelques lectures qui prolongent la réflexion sur la conception d'API — moins sur la syntaxe, plus sur les intentions derrière les choix.
 
- [**RESTful API Design: nouns are good, verbs are bad**](https://cloud.google.com/blog/products/api-management/restful-api-design-nouns-are-good-verbs-are-bad) (Google Cloud) — le billet classique sur la différence entre nommer des ressources et nommer des actions, et pourquoi le premier produit des API prévisibles.
- [**REST APIs must be hypertext-driven**](https://roy.gbiv.com/untangled/2008/rest-apis-must-be-hypertext-driven) (Roy Fielding) — l'inventeur de REST explique, avec une certaine exaspération, ce que la plupart des « API REST » ratent : l'hypermédia comme moteur de l'état. Court, tranchant, fondateur.
- [**Roy Fielding's Misappropriated REST Dissertation**](https://twobithistory.org/2020/06/28/rest.html) (Two-Bit History) — une histoire fascinante de comment le mot « REST » a dérivé de sa définition d'origine. Utile pour comprendre pourquoi tant de débats sur « est-ce vraiment du REST » tournent en rond.
- [**REST, Explained for Beginners**](https://htmx.org/essays/rest-explained/) (htmx) — une relecture du chapitre 5 de la thèse de Fielding pour des développeurs non-académiques. Défend l'idée que l'hypermédia (HTML, justement) est plus proche de l'esprit REST que les API JSON modernes.
- [**Web API Design Best Practices**](https://learn.microsoft.com/en-us/azure/architecture/best-practices/api-design) (Microsoft Azure) — un guide complet et pragmatique sur la modélisation des ressources, le filtrage, la pagination et le versionnage. Une bonne référence à garder sous la main.
- [**Most RESTful APIs aren't really RESTful**](https://florian-kraemer.net/software-architecture/2025/07/07/Most-RESTful-APIs-are-not-really-RESTful.html) (Florian Krämer) — une discussion accessible de ce que le découplage client-serveur via l'hypermédia apporte réellement, au-delà du débat « verbes ou noms ».
Le fil rouge de toutes ces lectures rejoint celui de cet article : une API n'est pas une projection de votre base de données sur le réseau. C'est une *interface* — au sens fort du terme. Elle existe pour que d'autres puissent comprendre et utiliser votre système sans avoir à lire votre code. Conçue ainsi, elle raconte une histoire. Conçue comme un miroir de vos tables, elle force chaque client à devenir archéologue.
